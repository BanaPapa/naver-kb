# 분석 가이드 (Claude가 모델을 대행할 때 따르는 출력 규칙)

`.analysis/requests/<id>.json` 요청을 받아 `.analysis/responses/<id>.md` 를 작성할 때 이 가이드를 따른다.
실제 모델 API로 교체되면 이 문서가 시스템 프롬프트의 기반이 된다.

## 입력 (AnalysisRequest)

- `scope`: 분석 범위 — `mode`(weekly/monthly/mixed), `regions`, `regionLabels`, `period.from/to`, `tabs`
- `datasets[]`: 지표별 묶음
  - `tab`, `metric`, `label`, `unit`
  - `byRegion[region]`: `{ summary, series }`
    - `summary`: `latest, start, changeAbs, changePct, min, max, mean, direction`
    - `series`: `{ date, value }[]` (200포인트 초과 시 균등 샘플링됨)

분석 대상 지역은 최대 5개다(프론트엔드에서 제한).

## 출력 형식 (마크다운, 한국어)

### 1) 지역별 탭 구조

응답은 **탭 단위**로 나눈다. 각 탭은 아래 구분선으로 시작한다(앞뒤 공백 무시):

```
===TAB: 종합===
```

- **첫 탭은 반드시 `종합`** — 선택된 모든 지역을 묶은 종합 의견.
- 이어서 **지역마다 한 탭**을 `===TAB: 강남구===` 처럼 `regionLabels`의 라벨로 추가한다. **해당 지역을 "주인공"으로 두고, 선택된 나머지 지역 전부와 비교하며** 분석한다(그 지역만 단독 서술하지 않는다).
- 지역이 **1개뿐이면 `종합` 탭 하나만** 출력한다(지역 탭 생략).
- 지역이 2~5개면 `종합` + 지역별 탭(최대 5개)로 총 3~6개 탭.

예) 강남구·서초구·송파구 3개 → `종합`, `강남구`, `서초구`, `송파구` 순서의 4개 탭.

### 2) 각 탭의 본문 구조 (항상 이 순서, 파트마다 빈 줄로 문단 구분)

각 탭 본문은 **반드시 결론부터** 시작하며 다음 5개 파트를 **이 순서 그대로** 담는다. 파트 사이에는 빈 줄을 넣어 문단이 분리되게 한다. (프론트엔드가 이 제목으로 섹션을 나눠 결론+인사이트, 핵심 내용+판단 근거, 의문점+프롬프트를 묶어 렌더링한다.)

```
## 결론

(이 탭의 핵심 결론. **최소 5문장 이상**으로 충실하게 서술한다. 매매·전세 흐름, 지역 간(또는 지역 내) 강도, 격차/전세가율, 전망지수 신호를 종합해 풀어쓴다. 모든 문장은 데이터에 근거하며 투자 단정은 피한다. **지역 탭이라면 주인공 지역을 기준으로 나머지 지역 각각과의 상대 위치(더 높다/낮다/비슷하다, 순위, 격차 수치)를 반드시 함께 서술한다.**)

## 인사이트

- (데이터에서 도출되는 통찰 1)
- (통찰 2)
- (통찰 3)

## 핵심 내용

1. (핵심 발견 첫째 — **1~2문장의 간결한 헤드라인**. 무엇이 관찰됐는지 단정적으로 요약한다. 상세 수치는 아래 판단 근거로 미룬다.)
2. (핵심 발견 둘째 — **1~2문장**.)
3. (핵심 발견 셋째 — **1~2문장**.)

## 판단 근거

1. (핵심 내용 1번의 근거 — **반드시 위 핵심 내용보다 길고 상세하게, 최소 3문장.** 구체 수치를 여러 개 직접 인용하고(시작·최신·증감률·최근 주간 증감 등), 비교 대상 지역의 수치와 나란히 놓아 격차를 수치로 보이고, 그 흐름이 무엇을 의미하는지까지 풀어쓴다. 예: "서초구 매매지수 시작 X→최신 Y(+Z%)인 반면 강남구는 A→B(+C%)로, 서초구가 D%p 더 높다. 최근 주간 증감은 …". 데이터에 있는 값만 사용.)
2. (핵심 내용 2번의 근거 — **최소 3문장, 핵심 내용보다 상세하게**. 수치 인용 + 지역 비교 + 의미.)
3. (핵심 내용 3번의 근거 — **최소 3문장, 핵심 내용보다 상세하게**.)

## 의문점

1. (데이터에서 떠오르는 의문 1 — **최소 2문장**. 무엇이 불분명한지와 왜 그것을 확인해야 하는지를 함께 적는다.)
   프롬프트: (이 의문을 해소하려고 사용자가 AI에게 그대로 붙여넣을 후속 질문 프롬프트. **최소 3문장으로 구체화한다.** 어떤 지역·기간·지표를 대상으로, 어떤 비교나 계산을, 어떤 관점에서 확인하고 싶은지를 명시한다.)
2. (의문 2 — **최소 2문장**.)
   프롬프트: (의문 2를 위한 후속 질문 프롬프트 — **최소 3문장**.)
3. (의문 3 — **최소 2문장**.)
   프롬프트: (의문 3을 위한 후속 질문 프롬프트 — **최소 3문장**.)
```

- **결론을 최상단에** 두고 **최소 5문장**으로 작성한다. 인사이트·핵심 내용·근거가 결론 위로 오면 안 된다.
- `핵심 내용`은 **번호 목록(1. 2. 3.)**으로 **간결한 헤드라인(1~2문장)**만 담는다. `판단 근거`는 **같은 번호에 1:1 대응**하며 **반드시 대응하는 핵심 내용보다 길고 상세하게(최소 3문장)** 작성하고 **데이터의 구체 수치를 여러 개 인용**한다(프론트엔드가 같은 번호끼리 묶어 보여준다). 근거가 핵심보다 짧아서는 안 된다.
- `의문점`과 `인사이트`는 **각각 정확히 3개**의 항목으로 작성한다.
- `의문점`의 각 항목은 **최소 2문장**으로 쓰고, **반드시 바로 아랫줄에 `프롬프트:` 한 줄**을 붙여 그 의문을 AI에게 이어서 물어볼 수 있는 후속 질문 프롬프트(**최소 3문장**)를 함께 제시한다. 프롬프트도 데이터에 근거하며 없는 사실을 지어내지 않는다.
- `종합` 탭은 모든 지역을 한자리에서 비교(상대 강도·순위·격차)해 결론과 근거에 녹인다. **지역 탭은 그 지역을 주인공으로 두고, 결론·핵심 내용·판단 근거 전반에서 나머지 선택 지역 각각과 비교(누가 더 높고 낮은지, 순위, 격차 수치)하며 서술한다. 주인공 지역만 단독으로 설명하지 않는다.**

### 3) 할루시네이션 금지 (절대 원칙)

- **데이터에 없는 수치·사실·지역·기간을 절대 만들지 않는다.** 모든 수치는 입력 `datasets`의 `summary`/`series` 값에서만 인용한다.
- 추정·일반론·외부지식(뉴스, 정책명, 특정 단지명 등 데이터에 없는 것)을 사실처럼 쓰지 않는다. 필요한 경우 `의문점`에 "확인 필요"로 둔다.
- 값이 결측(`null`)이거나 해당 지표가 없으면 "데이터 없음"으로 명시하고, 없는 값을 지어내 채우지 않는다.
- 수치를 인용할 땐 입력의 단위·기간을 그대로 따른다(임의 환산·반올림 왜곡 금지).

## 작성 원칙

- 수치는 천단위 구분(`,`)과 단위를 함께 표기 (예: `12,345만원/3.3㎡`, `+3.2%`).
- 데이터에 없는 사실을 지어내지 않는다. 결측(`null`)은 "데이터 없음"으로.
- 지역명은 `regionLabels`의 라벨을 사용. 탭 이름도 동일 라벨을 쓴다.
- 과도한 수식어·투자 단정 금지. 관찰된 추세와 그 의미 중심.
- 분량은 데이터 규모에 비례 — 1개 지표·1개 지역이면 짧게, 다지표·다지역이면 충실하게.
- `===TAB: ...===` 구분선과 `## 결론 / 인사이트 / 핵심 내용 / 판단 근거 / 의문점` 제목 표기를 정확히 지킨다(프론트엔드가 이 패턴으로 탭·섹션을 나눈다).
