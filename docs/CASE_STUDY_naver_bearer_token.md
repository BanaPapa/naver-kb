# 사례 연구: "빌라만 401 에러" — API 인증의 정체를 추적하기

> 학습·강의용 문서. 실제로 일어난 디버깅을 **재현 가능한 방법론**으로 정리했다.
> 코드를 고치는 법보다 **어떻게 원인에 도달했는가**가 핵심이다.
>
> 날짜: 2026-05-29 / 무대: 네이버 부동산 크롤러

---

## 0. 30초 요약

- **증상**: 빌라·단독/다가구를 검색하면 401, 아파트·오피스텔은 정상.
- **표면 가설(틀림에 가까움)**: "빌라는 토큰이 필요한가 보다. 근데 왜 덜 중요한 빌라에만?"
- **진짜 원인**: 중요도와 무관. 두 상품군이 **다른 엔드포인트**를 친다. 하나는 인증 강제, 하나는 비강제.
- **해결**: 비밀키를 깨려다 실패 → 발상 전환 → **네이버 JS가 만든 진짜 토큰을 헤드리스 브라우저로 가로채 재사용**.
- **교훈**: "왜 이건 되고 저건 안 되나"는 거의 항상 **변하지 않은 것과 변한 것의 차이**에 답이 있다.

---

## 1. 문제 제기와 첫 함정

사용자 관찰:
> "빌라·단독·사무실을 검색할 때만 401. 아파트·오피스텔로 바꾸면 정상.
> 예전 매크로는 요청 URL의 상품코드만 바꿔도 잘 됐는데? 아파트가 더 중요한데
> 왜 덜 중요한 빌라에 토큰이 필요하다는 거지? 잘못 판단한 거 아냐?"

여기에 **두 개의 함정**이 있다.

1. **의인화 함정**: "중요한 것엔 인증, 덜 중요한 것엔 없음"은 사람의 논리지 시스템의 논리가 아니다.
   서버는 중요도가 아니라 **엔드포인트별 정책**으로 인증을 건다.
2. **권위 함정**: "이전 AI(소넷)가 틀렸다"는 심증. 옳고 그름은 **증거**로 가린다.

> 🎓 **원칙 1 — 증상을 기능 중요도로 설명하려는 충동을 의심하라.**
> 시스템은 "중요해서"가 아니라 "그 경로가 그렇게 구현돼서" 동작한다.

---

## 2. 결정적 증거는 이미 사용자 손에 있었다

사용자는 Fiddler로 4종(아파트·오피스텔·빌라·단독)의 실제 요청/응답을 캡처해 제공했다.
여기서 **"같은 것끼리 나란히 놓고 무엇이 다른지"** 만 보면 됐다.

| 상품 | 실제 호출 엔드포인트 | Authorization 헤더 | 결과 |
|------|---------------------|-------------------|------|
| 아파트 | `/api/complexes/single-markers/2.0` | (있어도) 검사 안 함 | 200 |
| 오피스텔 | `/api/complexes/single-markers/2.0` | (있어도) 검사 안 함 | 200 |
| 빌라 | `/api/articles` | **검사함** | 토큰 있으면 200 |
| 단독/다가구 | `/api/articles` | **검사함** | 토큰 있으면 200 |

핵심: **둘 다 같은 도메인(`new.land.naver.com`)이지만 경로가 다르다.**
지도 마커 API는 토큰을 안 보고, 매물 리스트 API는 본다.
빌라/단독은 "단지" 개념이 없어 마커 API를 못 쓰고 매물 API로 갈 수밖에 없다 —
네이버 UI에서도 탭이 갈리는("아파트·오피스텔" vs "빌라·주택") 이유와 같다.

> 🎓 **원칙 2 — 되는 것 vs 안 되는 것을 나란히 diff 하라.**
> 변수를 하나로 좁히면(여기선 "엔드포인트") 원인이 드러난다.

### "예전 매크로는 왜 토큰 없이 됐나"의 진실
매크로는 **캡처한 요청을 통째로 재생**하며 URL의 상품코드만 바꿨다.
그 캡처에는 이미 `Authorization: Bearer ...`가 들어 있었다.
토큰이 "필요 없었던" 게 아니라 **항상 같이 따라다녀서 의식할 필요가 없었던** 것.
실제로 사용자가 준 빌라 요청에도 Bearer 헤더가 박혀 있었고 200을 받았다.

> 🎓 **원칙 3 — "원래 잘 됐다"는 기억은 종종 "원래 그 조건이 항상 충족돼 있었다"는 뜻이다.**
> 사라진 전제를 찾아라.

---

## 3. 토큰의 정체를 해부하다 (JWT 디코드)

받은 토큰을 디코드하면:
```json
// header
{"alg":"HS256","typ":"JWT"}
// payload
{"id":"REALESTATE","iat":1780015581,"exp":1780026381}
```
- `exp - iat = 10800` → **정확히 3시간**.
- `id`는 고정값 `"REALESTATE"`, **사용자 식별 정보가 없다.**

이 두 사실의 함의가 크다:
- 로그인 자격증명이 아니라 **앱이 자기 자신에게 발급하는 단기 서명 토큰**이다.
- 즉 네이버 프론트엔드 JS가 비밀키로 HS256 서명해 만들어낸다. → **이론상 자동 생성 가능.**

> 🎓 **원칙 4 — 토큰을 만나면 디코드부터 하라.**
> 페이로드가 "누구인지(sub/user)"를 담으면 로그인 자격증명, "무엇을 위한 것(role/app)"만 담고
> 수명이 짧으면 앱 발급 서명 토큰일 가능성이 높다. 대응 전략이 갈린다.

---

## 4. 막다른 길도 데이터다: 비밀키 브루트포스

첫 해결 시도: "HS256 비밀키를 알아내 직접 서명하면 토큰을 무한 발급할 수 있다."
이런 클라이언트 발급 토큰의 비밀키는 짧은 경우가 많다. 그래서:

- 사용자가 준 **실제 토큰 2개**(서명 유효)를 오프라인 크래킹 대상으로 삼음.
  → 비밀키를 찾으면 두 토큰 모두에 대해 서명이 재현되는지로 **네트워크 없이 검증** 가능.
- 사전 공격 + 길이 1~6 / 소문자+숫자 전수(24코어 병렬, 수억 후보).

**결과: 실패.** 비밀키가 6자보다 길거나 문자셋이 넓다는 뜻.
6자 초과 전수는 수 시간~불가 → **이 길은 접는다.**

> 🎓 **원칙 5 — 실패한 시도도 탐색 공간을 좁힌다.**
> "비밀키는 짧지 않다"는 음성 결과가 다음 결정(브루트포스 포기)을 이끈다.
> 그리고 **검증 가능한 길**(오프라인 서명 재현)을 우선하라 — 샌드박스가 네트워크 차단돼 있어도 진척이 난다.

---

## 5. 발상의 전환: 비밀키 대신 결과물을 훔친다

비밀키를 모르면 직접 서명은 불가. 하지만 **네이버 JS는 매번 유효한 토큰을 만들어낸다.**
그렇다면 그 JS를 실제로 돌리고 **출력(토큰)만 가로채면** 비밀키가 필요 없다.

도구는 이미 프로젝트에 있었다 — `puppeteer`(설치돼 있으나 미사용).

```
헤드리스 크롬 + 사용자 쿠키
  → new.land/houses 로드 (네이버 JS가 정상 실행되어 토큰 생성)
  → page.on('request')로 /api/* 요청의 Authorization 헤더 관찰
  → 첫 Bearer 캡처 → 메모리 캐시(만료 3시간)
```

이 방식의 강점:
- 비밀키 불필요. **비밀키가 바뀌어도(rotation) 안전** — 항상 네이버 JS 최신 로직을 쓴다.
- 서명 검증 여부와 무관 — 진짜 토큰이므로 무조건 통과.

비용: 첫 발급만 브라우저 기동(~3~5초), 이후 3시간 캐시. 아파트/오피스텔은 이 경로를 안 타므로 영향 없음.

> 🎓 **원칙 6 — 비밀을 못 얻으면, 비밀을 쥔 주체를 도구로 부려라.**
> 키를 역공학하는 대신, 키를 가진 시스템(네이버 JS)을 실행해 결과만 취한다.
> "재구현"보다 "재사용"이 거의 항상 더 견고하다.

---

## 6. 검증: "될 것 같다"가 아니라 "됐다"

추측으로 끝내지 않는다. 사용자의 실제 쿠키로 토큰 발급기를 **라이브 실행**:

```
[naver-token] 새 토큰 발급 성공 (만료까지 ~180분)
TOKEN: eyJ...   ← iat가 방금 시각. 네이버가 이 세션용으로 갓 발급한 진짜 토큰.
```

헤드리스 크롬이 네이버 JS를 실제 구동 → 새 토큰 생성 → 우리가 가로채기 성공.
(이 토큰이 `/api/articles`에 200을 주는 최종 확인은 샌드박스 IP의 레이트리밋(429) 때문에
같은 환경에서 못 했고, **실사용 환경에서 사용자가 확정**했다.)

> 🎓 **원칙 7 — 검증할 수 없는 수정은 미완성이다.**
> "should work"는 금지어. 라이브 실행 로그 같은 **관측 증거**를 남겨라.
> 환경 한계로 못 미치는 마지막 한 칸은 정직하게 명시하고 누가 어떻게 확정하는지 적는다.

## 7. 방법론 체크리스트 (재사용용)

비슷한 "왜 이건 되고 저건 안 되나" 문제를 만나면:

1. **증상 고정**: 정확히 무엇이, 언제, 어떤 에러코드로 실패하나? (401=인증, 403=권한, 429=레이트, 400=요청형식)
2. **나란히 diff**: 되는 케이스와 안 되는 케이스의 요청을 통째로 비교. 변수를 하나로 좁힌다.
3. **변하지 않은 전제 찾기**: "원래 됐다"면 무엇이 항상 충족돼 있었나?
4. **토큰/시크릿은 디코드**: 정체(자격증명 vs 앱 서명)와 수명을 먼저 파악.
5. **검증 가능한 길 우선**: 네트워크가 막혀도 진척 나는 오프라인 검증부터.
6. **막다른 길로 공간 좁히기**: 실패 결과를 다음 결정의 근거로.
7. **역공학보다 재사용**: 비밀을 쥔 주체를 실행해 결과를 취하는 길이 보통 더 견고.
8. **라이브로 검증**: 관측 증거를 남기고, 못 미친 한 칸은 정직하게 핸드오프.
